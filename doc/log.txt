
GNSS RTK 追踪日志系统设计方案
版本：1.0
状态：定稿
适用范围：Java RTK/PPP 解算库的追踪日志模块

一、设计目标与范围
1.1 核心目标
当解算结果出现异常（如模糊度从3变成5、位置跳变、固定失败）时，能够完整还原每个历元从输入到输出的全过程，精准定位问题发生在哪个环节。

1.2 设计原则
原则	说明
阶段固定	7阶段逻辑架构固定，所有历元走相同流程
输出可配	通过位掩码控制阶段开启/关闭，默认全关
库内构造	日志内容由库内部按规格构造，调用方只负责接收
零阻塞	回调接口必须快速返回，不阻塞解算线程
纯文本	所有输出为管道符分隔的键值对，便于解析
1.3 范围边界
库负责	调用方负责
定义 TraceControl 参数结构	创建 TraceControl 对象并传入
定义 TraceCallback 回调接口	实现 TraceCallback 接口
算法执行中按阶段构造日志字符串	决定日志落地方式（文件/MQTT/队列）
判断全局开关、阶段开关、采样控制	保证回调实现线程安全、非阻塞
格式化数值（Locale.US）	日志文件的轮转、清理、归档
异常时记录错误追踪	日志的采集、检索、展示

2.2 异常分支处理
所有异常触发逻辑（周跳、粗差、重初始化）全部收口在阶段2，通过以下字段显性体现：

字段	含义	溯源价值
slip=1	该卫星发生周跳，模糊度在本阶段被重置	解释为何某卫星模糊度突变为初始值
rejc>=2	该卫星粗差累计超限，触发状态重置	解释为何滤波器被强制重初始化
global=1	全局状态重置	解释为何所有卫星模糊度同时变化
溯源规则：若模糊度跳变，优先检查阶段2的 slip 和 reset 字段，区分“主动重置”和“滤波计算导致跳变”。

三、参数接口定义
3.1 TraceControl 结构
java
public class TraceControl {
    // 全局总开关，false则所有阶段都不输出，不检查任何后续条件
    public boolean enabled = false;

    // 阶段位掩码，控制哪些阶段开启（仅在enabled=true时生效）
    // 使用 STAGE_INPUT, STAGE_SATPOS, STAGE_UDSTATE, STAGE_DDRES, STAGE_FILTER, STAGE_LAMBDA, STAGE_RESULT
    public int stages = 0;

    // 内容标志位掩码，控制阶段内详细程度（仅在enabled=true时生效）
    // 使用 CONTENT_RESIDUAL_ONLY, CONTENT_H_MATRIX, CONTENT_SUMMARY_ONLY
    public int contentFlags = 0;

    // 最多记录多少个历元，0=不限（仅在enabled=true时生效）
    public int maxEpochs = 0;

    // 采样间隔：1=每个历元都记，10=每10个历元记一次（仅在enabled=true时生效）
    public int samplerate = 1;

    // 只记录这些卫星的细节（PRN号数组），空=全部（仅在enabled=true时生效）
    public int[] targetSats = new int[0];
}

3.3 TraceCallback 回调接口
java
public interface TraceCallback {
    /**
     * 接收追踪日志内容。
     *
     * 重要约束：
     * 1. 此方法由解算线程同步调用，必须快速返回，不得执行阻塞I/O
     * 2. 实现必须是线程安全的
     * 3. 推荐实现方式：将内容放入BlockingQueue，由独立消费者线程异步落盘
     *
     * @param content 格式为 "TRACE|阶段标识|键1=值1|键2=值2|..." 的字符串
     */
    void onTrace(String content);
}
四、统一输出格式规范
4.1 基本格式
text
TRACE|{phaseId}|gpst={gpst}|epoch={epoch}|{field1}={value1}|{field2}={value2}|...
4.2 字段规范
规则	说明
分隔符	管道符 |
键值对	key=value，等号前后无空格
数值格式	必须使用 String.format(Locale.US, "%.xf", value)，禁止依赖系统默认Locale
空值	输出 null 字符串，不省略字段
数组	多个值用逗号分隔，如 pairs=G01-G03,G01-G05
多行	同一阶段的多行输出，每行独立调用 onTrace()
4.3 公共字段
每个阶段输出的每一行都包含以下公共字段：

字段	类型	说明
gpst	double	GPS周秒（精度0.001s）
epoch	int	历元序号（从0开始）

阶段标识：STAGE1

输出行（每颗卫星一行）：

text

pos_y	double	卫星ECEF Y坐标（米），保留3位

阶段标识：STAGE2

输出行1（先验位置）：

字段	类型	说明

text
TRACE|STAGE2_RESET|gpst=123456.000|epoch=42|global=0|partial=
或（有重置时）：

text

字段	类型	说明
global	int	全局重置标志（0=无，1=发生）
partial	string	部分重置的卫星列表，逗号分隔，无重置时为空字符串
输出规则：

CONTENT_SUMMARY_ONLY 模式下，只输出 STAGE2 位置行，不输出逐卫星的 STAGE2_BIAS

溯源优先级：

若某卫星 slip=1 → 该卫星模糊度在本阶段被重置 → 解释跳变

若 global=1 → 全局状态重置 → 解释所有卫星同时跳变

若 partial 非空 → 这些卫星被重置 → 解释部分卫星跳变

六、判断流程与过滤规则
6.1 完整判断链（按优先级排列）
text
if (!traceControl.enabled) return;                    // ① 全局开关，最先判断
if (epoch >= traceControl.maxEpochs && maxEpochs > 0) return;
if (epoch % traceControl.samplerate != 0) return;
if ((traceControl.stages & stageBit) == 0) return;    // ② 阶段开关在全局之后
// 构造日志内容并回调
callback.onTrace(content);
优先级说明：

enabled=false 时，后面的 stages、maxEpochs、samplerate、targetSats 全都不需要检查

位运算虽轻量，但高频调用下累积开销不可忽略

语义清晰：enabled 是总闸，stages 是分路开关

6.2 卫星过滤
targetSats 不为空时：

阶段0的 STAGE0_SAT 行：只输出在 targetSats 中的卫星

阶段1的 STAGE1 行：只输出在 targetSats 中的卫星

阶段2的 STAGE2_BIAS 行：只输出在 targetSats 中的卫星

阶段3的 H 矩阵和残差：不按卫星过滤（因为双差是组合量，单独过滤某颗卫星会破坏完整性）

阶段5的模糊度：只输出在 targetSats 中的卫星

6.3 异常保护
java
try {
    String content = buildTraceContent(...);
    callback.onTrace(content);
} catch (Exception e) {
    // 日志回调异常不影响解算
    // 可选：记录一条内部错误日志
}
七、Java 实现约束
7.1 浮点数格式化（硬性要求）
所有数值输出必须使用：

java
String.format(Locale.US, "%.3f", value)
严禁使用：

java
Double.toString(value)           // 格式不可控
String.format("%.3f", value)     // 依赖系统默认Locale
new DecimalFormat("0.000")       // 依赖系统默认Locale
原因：欧洲语言环境（德语、法语、俄语）下，String.format 可能输出 12,34 而非 12.34，导致下游解析器按逗号分割字段时数值解析错误。

7.2 数组边界检查
所有数组访问前必须检查边界：

java
if (index < 0 || index >= array.length) {
    callback.onTrace("TRACE|ERROR|msg=index out of bounds|...");
    return defaultValue;
}
重点关注：

卫星号索引：sat - 1 必须 ≥ 0 且 < MAXSAT

模糊度索引：IB(sat, freq) 返回值必须 < nx

残差向量：v[i] 的 i 必须 < nv

7.3 回调非阻塞要求
TraceCallback.onTrace() 由解算线程同步调用，调用方必须保证实现快速返回。

推荐实现模式：

java
class AsyncTraceCallback implements TraceCallback {
    private final BlockingQueue<String> queue = new LinkedBlockingQueue<>();

    public AsyncTraceCallback() {
        new Thread(() -> {
            while (true) {
                String content = queue.take();
                // 异步落盘
            }
        }).start();
    }

    @Override
    public void onTrace(String content) {
        queue.offer(content);  // 非阻塞，立即返回
    }
}
库侧不做以下事情：

不检查回调是否阻塞

不设置超时

不提供日志队列（由调用方管理）

库侧仅做以下保护：

在回调外包一层 try-catch，确保回调异常不影响解算

记录回调异常到内部错误日志（如果有）

八、使用示例
8.1 调用方完整示例
java
// 1. 创建追踪控制参数
TraceControl ctrl = new TraceControl();
ctrl.enabled = true;
ctrl.stages = TraceControl.STAGE_DDRES | TraceControl.STAGE_FILTER | TraceControl.STAGE_RESULT;
ctrl.contentFlags = TraceControl.CONTENT_RESIDUAL_ONLY;
ctrl.maxEpochs = 100;
ctrl.samplerate = 1;
ctrl.targetSats = new int[]{1, 3, 5};  // G01, G03, G05

// 2. 实现异步回调
TraceCallback callback = new AsyncTraceCallback();

// 3. 注入到解算器
solver.setTraceControl(ctrl, callback);

// 4. 开始解算
solver.process(epochData);
